Contribute Unsolved Problems as Benchmarks

If an automated reasoning system fails to solve your input, contributing it as a benchmark helps the whole research community.

Welcome, and thank you for helping! üíõ

Sometimes automated reasoning systems struggle ‚Äî and that‚Äôs completely okay! If your input doesn‚Äôt get solved, contributing it as a benchmark is an incredibly generous way to help researchers and practitioners understand real-world challenges and make solvers better for everyone (SAT, PBO, MAXSAT, #SAT, SMT, ATP).

This page explains practical steps, what to include, and community-accepted formats for each subfield.

Why your contribution matters

  • Benchmarks drive progress ‚Äî they reveal weaknesses and inspire algorithmic advances.
  • Public benchmarks enable reproducible evaluation and fair comparisons.
  • Real-world, unsolved instances are often more valuable than synthetic ones.

üß© What to include with a contributed benchmark

  1. The instance file in the standard format for the community (examples below).
  2. Minimal reproduction: the exact input you gave to the solver (no hidden preprocessing unless documented).
  3. Solver invocation: command line, solver version, options, timeout, hardware used.
  4. Observed behaviour: timeout, wrong result, crash, memory error, indefinite run.
  5. Expected/desired outcome if known (satisfiable/unsatisfiable/counterexample/proof).
  6. Metadata: origin, license, short description, tags (optimization / combinatorics / verification), difficulty estimate.
  7. Optional: verification artifacts (proof traces, logs), model witness, or expected cost/value for optimization problems.

üìö Community-specific formats & clear statements

üî£ SAT (Boolean satisfiability)

Why it helps: SAT is a core technology used in scheduling, security, hardware and more. Hard cases move the whole field forward!

Clear statement: For SAT, submit instances in DIMACS CNF format (plain .cnf). Provide the exact file you fed the solver, your solver command line, timeout, and the solver's result (SAT/UNSAT/TIMEOUT/CRASH).

c Example comment lines start with 'c'
p cnf 6 9
1 -3 0
... (clauses) ...

üìä PBO (Pseudo-Boolean Optimization)

Why it helps: Optimization problems show where solvers struggle to find the best solution, not just any solution.

Clear statement: For PBO, provide the instance in the standard .opb (OPB) format or other community-accepted formats. Include the optimization objective and any assumptions used. State the expected optimality (if known) or that the optimum is unknown.

* #variable= 5 #constraint= 3
+1 x1 -2 x2 +3 x3 <= 4;
min: 2 x1 + 3 x2 + 1 x3;

üéØ MAXSAT

Why it helps: MAXSAT benchmarks highlight how solvers balance many conflicting goals.

Clear statement: For (Weighted) MAXSAT, use WCNF or CNF with soft/hard annotations. Provide weights, whether the instance is partial or weighted, and any target optimum if known.

p wcnf 4 6 10
1 1 -3 0
10 2 -4 0  (weight 10 soft clause)

üßÆ #SAT (Model counting / Weighted model counting)

Why it helps: Counting solutions pushes reasoning into probabilistic and AI applications.

Clear statement: For #SAT, submit the CNF instance, and indicate whether the count should be exact or approximate and whether it's weighted. If available, supply the known count or bounds, and the intended variable set to count over.

‚öôÔ∏è SMT (Satisfiability Modulo Theories)

Why it helps: SMT benchmarks test reasoning with arithmetic, bitvectors, arrays ‚Äî essential in software and hardware verification.

Clear statement: For SMT, use SMT-LIB v2 format (.smt2). Declare logic (QF_BV, QF_UF, QF_LIA, etc.) and provide the exact file and invocation. If the formula is intended for satisfiability or for producing a model/proof, state that explicitly.

; example.smt2
(set-logic QF_BV)
(declare-fun x () (_ BitVec 32))
(assert (= x #x00000000))
(check-sat)
(get-model)

üìê ATP (Automated Theorem Proving)

Why it helps: Testing ATP systems keeps mathematical reasoning systems trustworthy and ambitious.

Clear statement: For ATP (first-order logic, higher-order logic), use the TPTP format (.p, .tptp) and clearly label conjectures, axioms, and theory language (FOF, CNF, THF). Indicate expected result (theorem / countersatisfiable) and supply any existing proofs/traces if available.

% File: example.p
fof(ax1, axiom, (p(a))).
fof(conj, conjecture, (exists X: p(X))).

‚úÖ Practical checklist to attach to a submission

  • Instance file (in community format)
  • Description, a brief description what the benchmarks is about, i.e., what it models, or even a link to paper or some other medium with more details
  • Solver command used
  • Timeout & machine
  • Metadata (origin, license)
  • Optional proof/model/logs

üì¨ Where to submit

Different communities have different preferred channels. Typical places to submit:

  • Community benchmark repositories (e.g., SATLIB, SMT-LIB repository, TPTP for ATP)
  • Project repositories and issue trackers (solver project on GitHub/GitLab) ‚Äî open an issue or pull request with the instance and metadata
  • Conference or competition submission tracks (e.g., SAT/SMT/MAXSAT competitions) ‚Äî consult the competition's rules for submission format and deadlines
  • Community mailing lists and forums ‚Äî announce new, interesting unsolved instances there

If you are unsure where to send it, a GitHub repository with a permissive license + an issue describing the instance is a good starting point.

üîê Licensing, privacy & sensitive data

Please ensure you are allowed to share the instance. If it contains sensitive or proprietary data, either anonymize it or do not share it publicly. A simple permissive license recommendation for benchmarks is CC0 or explicit public-domain dedication when possible; otherwise choose a license that allows research use.
